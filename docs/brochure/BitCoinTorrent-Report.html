THE MODEL
Token = Licence
Every piece of content is a tokenised revenue stream. Hold tokens to serve content and earn revenue. No platform cut. No middleman.
Creator makes a video ↓ Mints BSV-21 token (e.g. 1,000,000 $EMPRESS tokens) ↓ Creator holds 100% of tokens = 100% of revenue + right to serve ↓ Viewers stream → pay per piece → 100% flows to token holders ↓ Creator sells tokens to seeders, investors, fans ↓ Seeders buy tokens = buy the licence to serve the content ↓ Revenue distributed proportionally via fan-out settlement TX
How Payments Work
Viewer opens payment channel → 1 on-chain funding TX Video plays → signed payment update per 256KB piece Each update fans out to N token holders proportionally Video ends → 1 on-chain settlement TX Outputs: one per token holder (amount = their share) Total: 2 on-chain transactions for any number of pieces

Payment Channels
Unidirectional nLocktime. Each payment update is a valid, broadcastable TX that fans out to all token holders. Only 2 on-chain TXs per session.
Content Tokenization
BSV-21 deploy+mint per video. Token ID links to content. Revenue address = token address. Holder snapshot via GorillaPool indexer API.
Fan-Out Settlement
Settlement TX has one output per token holder. Dynamic fee: 150 + N*34 sats. Recipient list from BSV-21 indexer snapshot.
Desktop App
Electron. Mac/Win/Linux builds. Settings (wallets, folders, pricing), My Library, file upload, tokenize modal. System tray seeding.

Our Design Decisions
No custom token
Pay in BSV sats. Content tokens are revenue rights, not payment currency.
Token = licence
Hold tokens to serve content. Seeders are investors with skin in the game.
No blockchain for discovery
Standard BitTorrent trackers. Blockchain only for payments + ownership.
100% to holders
No platform cut. No split. All revenue fans out to token holders atomically.
VOD only
Live P2P has physics problems. VOD streaming works perfectly with BitTorrent.
Fan-out settlement
The payment TX IS the distribution. No treasury. No custody risk.
